Consensia — Multi‑role AI Consensus for Software Engineering

Can a large language model be a trustworthy judge? Consensia studies stakeholder‑aware, evidence‑backed consensus across software engineering tasks — code, diagrams, and architectures — by orchestrating multiple expert personas (CTO, Architect, Developer, QA, SRE, Security, Product, Finance) and a meta‑evaluator.

Read the Aim See Status GitHub & Docs

Project Aim

We investigate whether an LLM can issue trustworthy, explainable judgments by comparing and scoring multi‑agent outputs. Consensia produces a final verdict with rationale, grounded in tool evidence (tests, linters, audits) rather than text alone.

Research question

Can an LLM reliably evaluate other LLMs’ outputs across software engineering domains, and explain the consensus?

Scope

Code generation & review, PlantUML/Mermaid diagrams, architectural decisions (ADRs) and trade‑off analysis.

Evidence

Unit tests, linters, security audits, diagram checks, and capacity estimates are normalized as structured signals for evaluation.

Outcome

Stakeholder‑aware verdicts (Adopt / Adopt with fixes / Rework) with confidence and highlighted trade‑offs.

How it works

Consensia coordinates multiple expert roles to generate artifacts and critiques, then tasks a meta‑judge to score candidates under a rubric. The verdict references specific evidence (e.g., tests passed, linter scores, threat findings).

1) Generate

Role agents (e.g., Developer, Architect) produce code, tests, diagrams, ADRs.

2) Verify

Sandboxed tools run: unit tests, linters, SCA, diagram lint, capacity checks.

3) Judge

A rubric‑guided LLM evaluates options using both text and tool signals.

4) Consensus

Stakeholder weights yield a final verdict with rationale and uncertainty.

Expert Roles

Each role applies a tailored rubric. Example dimensions: correctness, security, scalability, reliability, cost, product fit, clarity.

CTO

Strategy, risk, time‑to‑market.

Architect

Modularity, scalability, reliability, ADRs.

Developer

Correctness, readability, complexity, tests.

QA

Coverage, boundary cases, test plans.

SRE

SLIs/SLOs, capacity, failure modes, ops.

Security

Threat model, data classes, mitigations.

Product

Requirements fit, UX impact, scope.

Finance

Cloud/run cost and licensing trade‑offs.
